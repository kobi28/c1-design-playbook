Problem Statement

Before you design anything, get crystal clear on what you're solving. A sharp problem statement stops you from building the wrong thing.

Fill-in Template

The One-Liner Formula

Complete this sentence as a team. Write it on a sticky note. Everything you build should trace back to it.

[Type of user] needs a way to [do something] because [reason / pain point].

Example: "Remote team leads need a way to check project status in under 30 seconds because switching between five tools wastes 20 minutes every morning."

AI Prompt

Sharpen Your Problem with AI

If your problem feels vague, paste this into Claude or ChatGPT to tighten it up.

⌘
Copy this prompt → Paste into AI
I'm building a product in a 2-day hackathon. Here's my rough problem: "[paste your rough problem here]" Help me sharpen this into a clear problem statement using this format: [User type] needs a way to [action] because [pain point]. Give me 3 versions — one broad, one narrow, one surprising. For each, tell me who exactly the user is and what success looks like for them.
★ Pro Tip

Pick the narrowest version of the problem. In a hackathon, a focused product that does one thing brilliantly beats a vague product that does five things badly. You can always expand scope later — you can never get back wasted time.

Design

Solutioning with AI

This is where AI becomes your design partner. You don't need Figma. You don't need a design background. You need good prompts. Here's exactly what to ask.

AI Prompt

Generate User Flow

Start with how the user moves through your product, not what it looks like. Flow first, visuals later.

⌘
Copy this prompt → Paste into AI
I'm designing a product for this problem: "[paste your problem statement]" Map out the simplest user flow — from the moment the user opens the product to the moment their problem is solved. Rules: - Maximum 5–7 screens/steps - Each step should be one clear action - Name each screen (e.g., "Dashboard", "Upload", "Results") - Show me the happy path only — no edge cases yet Format as a numbered list with arrows showing the flow.
★ Pro Tip

If AI gives you more than 7 steps, your product is too complex for a hackathon. Ask it: "This is too many steps. Combine or cut until it's 5 screens maximum. What can I remove without breaking the core experience?"

AI Prompt Think Before You Build

Brief Each Screen

This prompt creates a detailed brief for each screen. Think of it like giving a builder an architect's drawing before they start construction — without it, they're guessing.

Why Not Skip Straight to Code?

When you ask AI to generate a full screen from a vague description, it has to figure out what to build and how it looks at the same time — and it does both badly. This prompt separates those decisions. You tell AI what goes on the screen. Then in the visual UI prompt, it focuses entirely on making it look and work right. The result is dramatically better, and you'll spend less time going back and forth fixing things.

Run this for each screen from your user flow. It takes about 2 minutes per screen.

⌘
Copy this prompt → Paste into AI
Now design the [screen name, e.g., "Dashboard"] screen for this product. Context: - Problem: "[problem statement]" - What the user just did: [previous step] - What they need to do here: [action on this screen] - What happens next: [next step] Design system — you MUST follow our internal design system "Pokedesk": [paste the relevant Pokedesk tokens/specs here — colours, typography, spacing, component names, button styles, etc. Even a screenshot of the guidelines page works.] Give me: 1. A layout description (what goes where — top, middle, bottom) using Pokedesk components where they exist (e.g., use the Pokedesk button, card, input, and nav components rather than inventing new ones) 2. The main UI elements — map each to its Pokedesk component name if possible 3. Exact text for headings, buttons, and labels 4. One micro-interaction that makes it feel polished (e.g., animation, loading state, confirmation) If a component I need doesn't exist in Pokedesk, flag it and suggest a design that feels consistent with the system. Keep it simple. This is a hackathon — no unnecessary features.

What you'll get back: A plain-English description of your screen — what goes where, which components to use, and what the copy says. It's not code yet. Save this output — you'll paste it into the "Generate the Visual UI" prompt to generate the actual UI.

★ Pro Tip — Feeding Pokedesk to AI

AI can't access Pokedesk directly, so you need to give it context. The easiest ways: (1) Copy-paste the colour tokens, type scale, and spacing values from the Pokedesk docs straight into your prompt. (2) Take a screenshot of an existing Pokedesk screen and upload it — AI can match the visual style. (3) Paste the CSS variables or Tailwind config from Pokedesk. The more you give it, the closer the output will match. Even a quick list like "Primary: #1A73E8, Border radius: 8px, Font: Inter, Spacing unit: 8px" makes a massive difference.

⚠ Why Pokedesk Matters

Using Pokedesk isn't just about looking consistent — it saves you time. Instead of debating colours, fonts, and button styles, you use what already exists. That's fewer decisions, faster builds, and a product that looks like it belongs to our ecosystem. Judges will notice.

Fake the data.
★ Best Practice — The Layered Prompt

Don't ask AI to design the whole app at once. Go in layers: flow → structure → visual → polish. Each prompt builds on the last. This gives you far better results than dumping everything in one giant prompt. Think of it like talking to a junior designer — give clear, specific instructions one step at a time.

Design Crit

Review — Is This Actually Good?

Before you polish anything, pause and check whether the design actually works. AI can generate beautiful screens that completely fail the user. This crit catches those problems before you waste time polishing the wrong thing.

The 6-Point Gut Check

As a team, open your screen and answer each question honestly. If you answer "no" to any of them, fix it before moving on.

1

Can you tell what this product does in 5 seconds?

Cover everything except the main screen. Show it to a teammate for 5 seconds. If they can't say what it does — your hierarchy or messaging is off.

2

Is the primary action obvious?

Every screen should have ONE thing you want the user to do. Can you point to it immediately? If there are two things competing for attention, one needs to be demoted.

3

Does it follow Pokedesk?

Check colours, fonts, spacing, and component styles against Pokedesk. AI often drifts from the design system — especially on border-radius, button styles, and colour shades. Compare side by side.

4

Is there anything you can remove?

Look at every element on screen and ask: "If I deleted this, would the user still succeed?" If yes — delete it. Less is almost always better in a hackathon.

5

Does the flow make sense end to end?

Click through every screen in order. Does each step feel like the logical next thing? Are there any dead ends or confusing jumps? The story should feel inevitable.

6

Can everyone use it?

Accessibility isn't a nice-to-have — it's how you make sure your product actually works for everyone. Even seasoned designers miss this. Do a quick check: Is your text readable against its background? (aim for 4.5:1 contrast ratio). Can you navigate the whole screen using only a keyboard — Tab, Enter, Escape? Do images and icons have text alternatives? Are clickable areas large enough to tap comfortably (at least 44×44px)? Does the design still make sense without colour alone (think colour blindness)?

Accessibility

The Hackathon A11y Cheat Sheet

You don't need to be an accessibility expert. You just need to not ship the five most common mistakes AI-generated UIs make. Here's the quick-fix list:

Colour Contrast

AI loves light grey text on white. Check yours at webaim.org/resources/contrastchecker — body text needs a 4.5:1 ratio minimum. If it fails, just make the text darker.

Keyboard Navigation

Put your mouse away. Press Tab through the page. Can you reach every button, link, and input? Can you tell which element is focused? If not, add :focus-visible styles and check your tab order.

Text Alternatives

Every image needs an alt attribute. Every icon-only button needs an aria-label. If a screen reader can't tell what something is, neither can a lot of your users.

Touch Targets

Buttons and links should be at least 44×44 pixels. AI tends to generate tiny icon buttons — make them bigger. Your thumb isn't a pixel-precise pointer.

Don't Rely on Colour Alone

If red means "error" and green means "success", add an icon or text label too. About 8% of men are colour blind — that's likely someone on your hackathon floor right now.

Crit Prompt

AI Design Review

Use AI as a second pair of eyes. Paste your screen code and let it find what you might have missed.

⌘
Copy this prompt → Paste into AI
You are a senior product designer reviewing a hackathon prototype. Be honest and constructive — like a design crit, not a compliment session. Here's the screen I built: [paste your HTML/code or describe the screen] The product solves this problem: "[problem statement]" Review it against these criteria and give me a score (pass / needs work / fail) for each: 1. CLARITY — Can a new user understand what this does and what to do first within 5 seconds? 2. FOCUS — Is there one clear primary action per screen, or are things competing for attention? 3. CONSISTENCY — Does the UI feel like it belongs to one coherent design system? Are colours, fonts, spacing, and components consistent? 4. SIMPLICITY — Is there anything that can be removed without hurting the core experience? 5. FLOW — Does the screen-to-screen journey feel logical and complete? 6. ACCESSIBILITY — Check: colour contrast (4.5:1 minimum for text), keyboard navigability (can you Tab through everything?), text alternatives on images/icons, touch targets at least 44×44px, and that colour isn't the only way information is conveyed. For any "needs work" or "fail" — give me ONE specific, actionable fix. Not theory — tell me exactly what to change and show me the corrected code.
★ Pro Tip — Pair Crit

The best hackathon teams crit each other. Find another team, swap screens for 2 minutes, and give each other honest feedback using the checklist above. Fresh eyes catch things you've gone blind to. You don't need a designer — you need someone who hasn't been staring at the same screen for an hour.

Build

Quick Prototype

Turn your AI-designed screens into something people can actually click through. Don't build a backend — fake everything. The goal is something that looks and feels real enough to test.

AI Prompt

Build a Clickable Prototype

⌘
Copy this prompt → Paste into AI
Turn these screen designs into a working, clickable prototype. Screens: [paste all your screen descriptions or code] Requirements: - Single HTML file with all screens as sections - Navigation between screens using JavaScript (show/hide) - Fake but realistic data - Transitions between screens (simple fade or slide) - Mobile-responsive - Include a clear "start" state This should feel like a real app someone can click through on a phone during a demo.
★ Pro Tip

Fake the AI. If your hackathon product uses AI, you don't need it working for the prototype. Hardcode the AI's response. Show what the output would look like. Judges care about the idea and the experience, not whether the API call is live.

⚠ Common Mistake

Don't spend time on auth screens. No one cares about login during a demo. Start your prototype at the first useful screen. If you need to show "logged in", just hardcode a name in the header.

Validate

Test It

Grab someone who isn't on your team. Show them the prototype. Shut up and watch. That's it. Three tests will reveal 80% of your design problems.

Script

The 3-Minute Test Script

Say this to your tester, word for word:

1

Set Up (30 seconds)

"I'm going to show you something we're building. I want you to think out loud — tell me what you see, what you'd click, and anything that confuses you. There are no wrong answers. You can't hurt my feelings."

2

Task (2 minutes)

"Imagine you're [describe the user]. Your goal is to [describe the main task]. Go ahead and try."

3

Observe (the whole time)

Don't help. Don't explain. Don't say "oh you need to click there." If they're stuck, ask: "What are you looking for?" Write down where they hesitate, what they misunderstand, and what they ignore.

★ Pro Tip

The "5-second test" — show someone the main screen for 5 seconds, then hide it. Ask: "What is this product? What would you do first?" If they can't answer, your design isn't communicating clearly enough.

Iterate

Learn & Loop Back

After testing, you'll have a short list of what's broken. Use AI to fix it fast — then test again. The more loops you do, the better your product gets.

Prioritise by impact — what's the smallest change that fixes the biggest problem? Then update the code.
Framework

The Quick Debrief

After each test round, answer these three questions as a team:

✓

What worked?

What did people understand immediately? What felt natural? Keep these things.

△

What confused people?

Where did they pause, squint, or click the wrong thing? These are your top-priority fixes.

?

What surprised us?

Did anyone use the product in a way you didn't expect? That's signal — it might reveal a better direction.

⚠ Remember

You don't need to fix everything. In a hackathon, fix the top 2–3 issues and move on. Perfection is the enemy of shipping. One tight loop of test → fix → test is worth more than four hours of polishing pixels nobody sees.
